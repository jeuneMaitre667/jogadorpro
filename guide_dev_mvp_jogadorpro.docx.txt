1.3 - Installation Git

📦 GIT (Version Control)

QU'EST-CE QUE C'EST?
Système de contrôle de version. Sauvegarde ton code, historique, backup.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
INSTALLATION - WINDOWS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ÉTAPE 1: Télécharger
□ Va sur: https://git-scm.com/download/win
□ Téléchargement auto démarre
□ Fichier: Git-2.x.x-64-bit.exe

ÉTAPE 2: Installer
□ Double-clique .exe
□ OPTIONS IMPORTANTES:
 
 Écran "Select Components":
 ☑ Git Bash Here
 ☑ Git GUI Here
 ☑ Associate .git* files
 
 Écran "Choosing the default editor":
 → Sélectionne "Use Visual Studio Code as Git's default editor"
 
 Écran "Adjusting PATH":
 → Sélectionne "Git from the command line and also from 3rd-party software"
 
 Écran "Choosing HTTPS transport":
 → "Use the OpenSSL library"
 
 Écran "Configuring line endings":
 → "Checkout Windows-style, commit Unix-style"
 
 Écran "Terminal emulator":
 → "Use MinTTY"
 
 Reste: Options par défaut OK

□ Clique "Install"
□ Attends 2-3 min
□ Finish

LINUX - Terminal natif
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

OUVRIR:
Ctrl + Alt + T

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
COMMANDES DE BASE (À CONNAÎTRE)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

pwd = Print Working Directory (où suis-je?)
ls = List (liste fichiers/dossiers)
cd = Change Directory (changer de dossier)
 Exemple: cd Documents
 Retour: cd ..
mkdir = Make Directory (créer dossier)
 Exemple: mkdir mon-projet
clear = Efface l'écran
code . = Ouvre VS Code dans dossier actuel

TEST RAPIDE:
1. pwd (voir où tu es)
2. ls (voir ce qu'il y a)
3. mkdir test-folder (créer dossier test)
4. cd test-folder (entrer dedans)
5. pwd (vérifier que tu es dedans)
6. cd .. (retour)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
VS CODE TERMINAL INTÉGRÉ (Recommandé!)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Meilleure façon: Terminal DANS VS Code!

OUVRIR:
□ VS Code → Menu "Terminal" → "New Terminal"
□ Ou: Ctrl + ` (backtick, à gauche du 1)

CONFIGURER DEFAULT:
Windows:
□ Ctrl + Shift + P
□ Tape: "Terminal: Select Default Profile"
□ Sélectionne "Git Bash"

Mac/Linux: Terminal natif OK par défaut

AVANTAGES:
✅ Pas besoin switcher fenêtres
✅ Même dossier que ton code
✅ Multiple terminaux possibles

FRONTEND (Ce que l'utilisateur voit)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

⚛️ REACT
Quoi: Librairie JavaScript pour créer interfaces
Pourquoi: La plus populaire (énorme communauté)
Analogie: Les briques Lego pour construire ton site

⏭️ NEXT.JS 14
Quoi: Framework React (React avec super-pouvoirs)
Pourquoi: 
 - SEO optimisé (Google indexe bien)
 - Performance excellente
 - Déploiement facile
 - Server + Client en un
Analogie: React = moteur de voiture, Next.js = voiture complète

🎨 TAILWIND CSS
Quoi: Framework CSS (styling)
Pourquoi:
 - Rapide à écrire
 - Looks professionnel
 - Responsive auto
 - Pas besoin écrire CSS from scratch
Analogie: Styles pré-faits comme templates Canva

🧩 SHADCN/UI
Quoi: Composants React beaux et accessibles
Pourquoi:
 - Copy-paste ready
 - Customizable
 - Looks moderne
 - Gratuit
Exemple: Buttons, cards, modals, forms déjà faits

📘 TYPESCRIPT (optionnel pour commencer)
Quoi: JavaScript avec types (plus safe)
Pourquoi: Évite bugs, meilleur dev experience
Note: On peut commencer en JavaScript pur, migrate après

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BACKEND (Logique + Base de données)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🔥 SUPABASE
Quoi: Backend-as-a-Service (tout-en-un)
Inclut:
 - PostgreSQL database (puissant, gratuit 500MB)
 - Authentication (signup/login)
 - API auto-générée (pas besoin coder!)
 - Storage (files/images)
 - Real-time (bonus)

Pourquoi Supabase vs autres:
✅ Gratuit tier généreux (500MB database, 50K users)
✅ PostgreSQL (meilleur que MySQL pour finance)
✅ API auto → développement 5x plus rapide
✅ Auth built-in → pas besoin coder login from scratch
✅ Open source → pas vendor lock-in

Alternative considérées:
- Firebase: Bon mais NoSQL (moins bon pour finance, relationships)
- Custom backend (Node + PostgreSQL): Plus de contrôle mais 3x plus long
- Prisma + PlanetScale: Excellent mais plus complexe setup

Verdict: Supabase = Best balance rapidité/puissance/coût

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

2.2 - Structure Base de Données

🗄️ SCHÉMA DATABASE (Tables & Relations)

Notre database PostgreSQL aura ces tables principales:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TABLE: users
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Colonnes:
- id (UUID, primary key)
- email (string, unique)
- full_name (string)
- created_at (timestamp)
- role (enum: 'user', 'admin', 'funded')
- kyc_status (enum: 'pending', 'verified', 'rejected')
- kyc_documents (jsonb) - stocker metadata documents

Relation: user → challenges (one-to-many)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TABLE: challenges
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Colonnes:
- id (UUID, primary key)
- user_id (UUID, foreign key → users)
- tier (enum: 'starter', 'pro', 'elite')
- price_paid (decimal)
- status (enum: 'active', 'passed', 'failed', 'completed')
- phase (integer: 1 ou 2)
- start_date (timestamp)
- end_date (timestamp)
- initial_balance (decimal) - ex: 10000
- current_balance (decimal)
- target_profit (decimal) - ex: 1000 (10%)
- max_daily_loss (decimal) - ex: 500 (5%)
- max_total_loss (decimal) - ex: 1000 (10%)
- created_at (timestamp)

Relation: challenge → bets (one-to-many)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TABLE: bets
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Colonnes:
- id (UUID, primary key)
- challenge_id (UUID, foreign key → challenges)
- user_id (UUID, foreign key → users)
- bet_type (string) - ex: 'Moneyline', 'Over/Under'
- sport (string) - ex: 'Football', 'Basketball'
- event_description (string) - ex: 'Flamengo vs Palmeiras'
- odds (decimal) - ex: 1.85
- stake (decimal) - montant parié
- potential_win (decimal) - gain potentiel
- result (enum: 'pending', 'won', 'lost', 'void')
- profit_loss (decimal) - P&L réel
- placed_at (timestamp)
- settled_at (timestamp, nullable)

Calculs automatiques:
potential_win = stake × odds
profit_loss = (stake × odds - stake) if won, else -stake

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TABLE: funded_accounts
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Colonnes:
- id (UUID, primary key)
- user_id (UUID, foreign key → users, unique)
- challenge_id (UUID, foreign key → challenges) - challenge qui a qualifié
- capital_allocated (decimal) - ex: 10000
- current_balance (decimal)
- total_profit (decimal) - cumulé
- profit_split (decimal) - % trader (ex: 0.80 = 80%)
- status (enum: 'active', 'suspended', 'terminated')
- activated_at (timestamp)

Relation: funded_account → payouts (one-to-many)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TABLE: payouts
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Colonnes:
- id (UUID, primary key)
- funded_account_id (UUID, foreign key)
- user_id (UUID, foreign key)
- amount (decimal) - montant payout
- trader_share (decimal) - 80% du profit
- platform_share (decimal) - 20% du profit
- status (enum: 'pending', 'processing', 'paid', 'failed')
- requested_at (timestamp)
- processed_at (timestamp, nullable)
- payment_method (string) - ex: 'PIX', 'Bank Transfer'
- payment_details (jsonb) - bank info, PIX key, etc

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TABLE: transactions
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Colonnes:
- id (UUID, primary key)
- user_id (UUID, foreign key)
- challenge_id (UUID, nullable)
- type (enum: 'challenge_purchase', 'payout', 'refund')
- amount (decimal)
- currency (string) - 'EUR', 'BRL'
- status (enum: 'pending', 'completed', 'failed', 'refunded')
- payment_provider (string) - 'stripe', 'mercado_pago'
- provider_transaction_id (string) - Stripe payment ID

🏗️ ARCHITECTURE APPLICATION

Comment tout s'assemble:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
COUCHES APPLICATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. FRONTEND (Client-side)
  React Components → affiche UI
  Next.js Pages → routing
  Tailwind → styling
  
2. API LAYER (Server-side)
  Next.js API Routes → endpoints custom
  Supabase Auto API → CRUD database

  PostgreSQL (Supabase) → stocke data
  
4. SERVICES EXTERNES
  Stripe → paiements
  Resend → emails

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
FLOW EXEMPLE: USER SIGNUP
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. User visite /signup
2. Next.js rend page signup
3. User entre email + password
4. Frontend → Supabase Auth API
5. Supabase:
  - Crée auth.users entry
  - Crée public.users entry
  - Envoie email vérification
6. User reçoit email
7. User clique lien → vérifié
8. Redirect dashboard

Temps total: <30 secondes

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
FLOW EXEMPLE: CHALLENGE PURCHASE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. User logged in, visite /pricing
2. User sélectionne tier (ex: Pro 89€)
3. Clique "Purchase"
4. Frontend → Next.js API route /api/create-checkout
5. API route → Stripe: créer checkout session
6. Stripe retourne URL checkout
7. Frontend → redirect Stripe checkout page
8. User entre carte bancaire
9. Stripe process payment
10. Si succès → Stripe webhook → notre API /api/webhooks/stripe
11. Webhook:
   - Vérifie signature
   - Crée entry dans challenges table
   - Crée entry dans transactions table
   - Update user
12. API → Resend: envoie email "Challenge Started!"
13. User redirect → /dashboard/challenges
14. Voit nouveau challenge actif

Temps: ~2 minutes

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
FLOW EXEMPLE: PLACE BET
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. User dans challenge dashboard
2. Clique "Place Bet"
3. Modal s'ouvre
4. User entre:
  - Event: "Flamengo vs Palmeiras"
  - Bet type: "Moneyline"
  - Odds: 1.85
  - Stake: 100€
5. Frontend calcule potential win: 185€
6. User confirme
7. Frontend → Supabase insert bet:
  {
    challenge_id: xxx,
    user_id: yyy,
    event_description: "Flamengo vs Palmeiras",
    odds: 1.85,
    stake: 100,
    potential_win: 185,
    result: 'pending'
  }
8. Supabase update challenge.current_balance -= 100
9. Frontend refresh → bet apparaît dans liste
10. User settle bet plus tard (won/lost)
11. System update P&L automatiquement

Temps: <1 minute

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
SÉCURITÉ (Row Level Security - RLS)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Supabase RLS = règles qui empêchent users voir data des autres

Exemple règles:
- users: Can only read/update OWN profile
- challenges: Can only see OWN challenges
- bets: Can only create/read bets for OWN challenges
- funded_accounts: Can only see OWN account
- admin: Can see EVERYTHING

SQL Example:
CREATE POLICY "Users can view own challenges"
ON challenges FOR SELECT
USING (auth.uid() = user_id);

→ Sécurité built-in database level! 🔒

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PERFORMANCE OPTIMIZATIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- Server-side rendering (Next.js): SEO + fast first load
- Static generation: Landing/pricing pages = HTML static
- API caching: Cache responses fréquentes
- Database indexes: Queries rapides
- Image optimization: Next.js Image component
- Code splitting: Load only needed code
- CDN (Vercel): Serve content from nearest location

Result: Page load <1s, interactions <100ms 🚀

import { createClient } from '@supabase/supabase-js'

const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL!
const supabaseAnonKey = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!

export const supabase = createClient(supabaseUrl, supabaseAnonKey)

import { supabase } from '@/lib/supabase/client'
import { useEffect, useState } from 'react'

export default function Home() {
 const [connected, setConnected] = useState(false)
 
 useEffect(() => {
   const testConnection = async () => {
     const { error } = await supabase.from('_').select('*')
     setConnected(!error || error.message.includes('does not exist'))
   }
   testConnection()
 }, [])
 
 return (
   <main className="p-24">
     <h1 className="text-4xl font-bold mb-4">JogadorPro</h1>
     <p>Supabase: {connected ? '✅ Connected!' : '❌ Not connected'}</p>

 )
}
